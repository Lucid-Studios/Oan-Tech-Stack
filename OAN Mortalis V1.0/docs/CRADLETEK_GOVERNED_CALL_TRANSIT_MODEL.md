# CradleTek Governed Call Transit Model

## Purpose

This note defines CradleTek as the governed call-transit spine of the active OAN runtime stack.

It exists to prevent the architecture from drifting into a false picture in which public host traffic, protected dispatch, return packets, collapse packets, or telemetry-significant crossings move sideways between layers without a lawful governance passage.

## Boundary

This note does not make CradleTek:

- the semantic mind of SLI/Lisp Engineered Cognition
- the sovereign identity author
- the final Sanctuary accountability judge
- the replacement for SoulFrame legality or Steward-guided adjudication

This note does define:

- what counts as a governed call
- what call formation requires
- what CradleTek must carry and encode
- what may never bypass the CradleTek transit spine
- how return and collapse traffic differ from ordinary service return

## Root Statement

CradleTek is the governed call-transit spine of the OAN runtime stack. All governance-relevant ingress, protected dispatch, return traffic, collapse traffic, and cross-boundary service transitions must pass through CradleTek for telemetry encoding, lifecycle control, and lawful routing.

## Governing Lines

> CradleTek is the runtime covenant spine through which stack traffic is made lawful.
>
> The membrane may judge legality, but it may not judge meaning.
>
> Transit is not authority, and carriage is not mutation.
>
> Sanctuary hosts the public-facing service demand; CradleTek forms the governed call; AgentiCore performs the protected work.
>
> No governance-relevant cross-boundary service traffic may bypass CradleTek.

## What Counts As A Governed Call

A governed call is any service transition whose formation, dispatch, return, or collapse matters to stack governance, telemetry, or protected-boundary law.

This includes:

- public-facing service ingress from Sanctuary
- protected dispatch into SoulFrame-mediated or AgentiCore-hosted service lanes
- target-runtime issue into SLI/Lisp cognition lanes
- return traffic carrying traces, residues, receipts, or conditioned derivatives
- collapse traffic carrying protected closure candidates
- cross-boundary transitions between Prime-facing and Cryptic-side service strata

This does not mean every internal method call in every component must route through CradleTek.

It does mean that every governance-relevant boundary crossing must do so.

## Call Formation

CradleTek participates in call formation before protected execution begins.

The governed call envelope should name at least:

- call origin
- caller posture
- mission or service scope
- target capability or lane
- Theater or locality posture where required
- protection class
- telemetry class
- return class
- collapse sensitivity
- refusal conditions

No call is lawful merely because a host surface requested it.

## Call Dispatch

Once formed, CradleTek governs dispatch.

Dispatch duties include:

- runtime and service selection
- protected-lane eligibility checks
- SoulFrame contextualization handoff where required
- AgentiCore service tenancy provisioning where required
- target-runtime capability checks
- lifecycle lease and fault posture
- telemetry envelope propagation

CradleTek therefore governs passage into Prime-side and Cryptic-side work without becoming the semantic owner of that work.

Governed call transit must therefore be read as carriage, encoding, and routing authority, not as mutation authority over the control surfaces touched by the call.

## Ordinary Return Traffic

Ordinary return traffic is the non-collapse return path for bounded service execution.

It may include:

- result packets
- symbolic traces
- residues
- conditioning receipts
- refusal receipts
- capability or posture reports

CradleTek must:

- normalize the return envelope
- preserve provenance and telemetry encoding
- route the packet to the next lawful consumer
- refuse direct public release when conditioning or admissibility law has not been satisfied

The first executable landing now makes the return path explicit with:

- actionable content handles
- governed request envelopes
- mutation receipts
- terminal `.hopng` evidence receipts

## Collapse Traffic

Collapse traffic is not identical to ordinary return traffic.

Collapse traffic carries protected closure material such as:

- return candidates
- protected residues
- self-bearing implications
- interlocutory `cGoA` candidates
- later adjudication candidates

CradleTek must therefore keep collapse routing distinct from ordinary result return while still serving as the governed transit spine for both.

## Anti-Bypass Law

The following must not bypass CradleTek:

- public host ingress into protected service lanes
- protected service dispatch into AgentiCore or target SLI/Lisp runtime lanes
- governance-relevant telemetry-significant transitions
- conditioned return to public or Prime-facing service surfaces
- collapse traffic toward `cGoA`, `cMoS`, Steward-guided adjudication, or later retained classes

The following may remain local and internal without independent CradleTek transit handling:

- purely internal function calls inside a single lawful component
- local computation that does not cross a governance-relevant boundary
- deterministic helper calls that neither change service posture nor cross protected boundaries

## Authority Split Around The Transit Spine

### Sanctuary

Sanctuary hosts the resident public-facing service surface and receives lawful outcomes back from CradleTek.

### CradleTek

CradleTek forms, encodes, routes, witnesses, and returns governed call traffic.

### SoulFrame

SoulFrame provides self-bearing contextualization, legality mediation, and return-path narrowing where self-bearing implications are involved.

### AgentiCore

AgentiCore is the protected operative pair that receives lawful protected dispatch and performs bounded runtime cognition.

### SLI/Lisp Runtime

The target SLI/Lisp runtime owns the semantic meaning of lawful migrated cognition lanes once dispatch reaches it.
