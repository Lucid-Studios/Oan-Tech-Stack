# SLI Lisp EC Engram Runtime Model

## Purpose

This note defines how engrams exist and operate in real time inside the SLI/Lisp Engineered Cognition environment.

It exists to prevent a weak reading of engrams as passive stored memory blobs after the stack has already crossed into live symbolic runtime participation.

This pass is about runtime life, not only storage ontology.

## Boundary

This note does not define:

- governance adjudication
- Prime publication law
- SoulFrame identity authorship
- Sanctuary accountability judgment
- full graphics-card harness implementation

This note does define:

- runtime engram kinds
- runtime states
- lawful live operations
- illegal operations
- trace and residue duties
- return and collapse eligibility at the runtime boundary

The stored structure versus admitted use versus live expression distinction is defined separately in:

- `docs/SKILL_ABILITY_TALENT_ENGRAM_ADMISSION_LAW.md`

## Root Principle

An engram in the SLI/Lisp EC runtime is a lawful symbolic-operational unit, not merely remembered content.

That means a live engram must be understood in at least five forms:

1. declared form
2. resident form
3. operative form
4. evidentiary form
5. collapse or return form

## Runtime Principle

An engram is not just remembered content. It is a lawful symbolic unit of runtime participation.

In runtime, an engram may therefore be:

- addressable
- typed
- activated
- composed
- compared
- cleaved
- residue-producing
- witnessable
- return-candidate bearing

This runtime role belongs to SLI/Lisp cognition, not to passive storage surfaces alone.

## Engram Kinds

The first runtime doctrine should distinguish kinds before implementation details.

Minimum lawful runtime classes:

- `PointerEngram`
  - a bounded addressing or reference carrier
- `ExpressionEngram`
  - a symbolic expression-bearing unit
- `RoleShellEngram`
  - a role, posture, or office-bearing symbolic unit
- `OperatorBondEngram`
  - a bond- or operator-related unit
- `WitnessEngram`
  - a comparison or preservation-bearing runtime unit
- `TransportEngram`
  - a carry-bearing unit downstream of lawful witness
- `ResidueEngram`
  - a useful remainder or unresolved remainder unit
- `ObstructionEngram`
  - a refusal, mismatch, or blocked-actualization unit
- `GovernanceCandidateEngram`
  - a runtime unit that may later be packetized for governance-facing treatment
- `CompositeEngram`
  - a braided or locally composed unit built from other bounded units
- `HotWorkingEngram`
  - an active runtime-only unit inside hot `c*` cognition
- `ReturnCandidateEngram`
  - a unit shaped for membrane handoff or collapse evaluation
- `StabilizedEngram`
  - a unit that has cooled enough to stand as a bounded stabilized surface

These are runtime classes, not yet public contracts.

## Runtime States

Every live engram should move through explicit runtime states rather than vague activation language.

Minimum state vocabulary:

- `Latent`
  - known but not resident in the active lane
- `Loaded`
  - present in runtime scope but not yet active
- `HotActive`
  - actively participating in bounded cognition
- `Braided`
  - locally composed with other units under lawful relation
- `Witnessed`
  - carrying explicit preservation and difference structure
- `TransportCandidate`
  - eligible for lawful carry only after witness law holds
- `Obstructed`
  - blocked by contradiction, incompatibility, or missing law
- `ResidueBearing`
  - carrying non-discarded useful remainder
- `ReturnCandidate`
  - eligible for membrane handoff
- `Deferred`
  - neither admitted nor discarded
- `ReEngrammitized`
  - lawfully re-shaped into a later engram-bearing form
- `Stabilized`
  - cooled enough for bounded stabilized treatment

No state should silently imply governance admission, Prime publication, or Sanctuary judgment unless a later governing lane says so explicitly.

## Minimum Runtime Cycle

The minimum live runtime cycle for a single engram unit should be:

1. intake or summon
2. type declaration
3. symbolic address formation
4. activation into hot working scope
5. operation or composition
6. trace and residue production
7. return-candidate shaping
8. membrane handoff eligibility
9. governance disposition
10. re-engrammitization, defer, or rejection

This is the first runtime loop to get right.

## Allowed Operations

The first operational law should name a bounded verb set.

Minimum lawful operations:

- `instantiate`
  - create a bounded runtime unit from a lawful source
- `bind`
  - attach the unit to runtime scope, role, locality, or relation
- `address`
  - form a stable symbolic address or handle
- `activate`
  - move the unit into hot working participation
- `uplift`
  - raise the unit into a more expressive local symbolic form without bypassing law
- `cleave`
  - separate a unit into lawful parts and residues
- `braid`
  - compose multiple units while preserving local distinctions
- `annotate`
  - attach lawful metadata, trace, or evidence
- `compare`
  - test the unit against another unit under explicit comparison law
- `witness`
  - name preserved invariants and lawful differences
- `residue-extract`
  - pull out useful remainder without silently discarding it
- `suspend`
  - take the unit out of hot active flow without declaring it resolved
- `rehydrate`
  - bring a latent or cooled unit back into bounded active scope
- `shape-return-candidate`
  - prepare a unit for membrane or collapse evaluation
- `return-to-membrane`
  - hand the unit out of hot runtime into the lawful membrane path
- `reject`
  - refuse a unit lawfully
- `defer`
  - preserve the unit without false resolution
- `re-engrammitize`
  - reshape the unit into a new lawful engram-bearing form under later authority

These operations are doctrine-first and should later map into explicit SLI op families or IR instructions.

## Illegal Operations

The runtime must also name illegal verbs or patterns explicitly.

Forbidden runtime moves:

- hot surface self-validation
- unreceipted cross-boundary mutation
- silent host fallback meaning substitution
- governance impersonation by SLI
- membrane impersonation by SLI
- untyped collapse
- raw Prime publication from hot runtime state
- direct GEL or custody mutation from runtime symbolic execution
- silent contradiction repair by local invention
- treating symbolic trace alone as equivalent to lawful engram stabilization

If the target runtime lane is migrated and unavailable, host execution of the same cognition semantics is also forbidden unless a separately labeled non-cognitive fallback path exists.

## Cross-Surface Law

Live engram runtime must preserve the hot/cool self-validation split.

That means:

- hot `c*` runtime engrams may reason, compose, and detect contradiction
- cooler stabilized surfaces remain validation-bearing where self, role, capability, or autobiographical truth is at stake
- unresolved contradiction becomes `Obstructed`, `ResidueBearing`, or `Deferred`

A hot runtime engram may not author self-validity by fluency alone.

## Trace And Witness Contract

Every lawful live engram operation should emit enough observability to prove the runtime did real work.

Minimum observability requirements:

- input declaration
- operation name
- runtime scope
- source handle or source reference
- resulting expression or transformed state
- residue produced
- witness or trace output
- admissibility or return status
- collapse disposition eligibility

Without this, runtime cognition drifts back into uninspectable naming.
