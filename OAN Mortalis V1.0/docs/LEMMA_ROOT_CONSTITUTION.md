# LEMMA_ROOT_CONSTITUTION

## Purpose

This document defines the root-core law for the canonical Atlas.

The key clarification is:

- surface words are not canonical root identity
- lemmas are canonical root identity

The Atlas therefore operates at the lemma layer, not the surface-token layer.

## Core Rule

For the active Atlas model:

- `PredicateRoot.Key` must be interpreted as the lemma identity
- lexical variants remain downstream forms
- prefix and suffix operators modulate the lemma core
- symbolic constructor bodies are built around lemma anchors

This keeps lexical inflection separate from semantic identity.

## Layer Distinction

The stack now distinguishes:

- tokens
  - transport or expression surface
- words
  - lexical variants and observed forms
- lemmas
  - semantic root anchors
- operators
  - lawful grammatical modulation
- Engrams
  - invariant meaning objects

The Atlas sits at the lemma layer.

## Lemma Identity Policy

Canonical root identity must satisfy:

- one stable lemma key per root
- one canonical atlas entry per lemma
- lexical variants normalize back to lemma identity
- constructor operators attach to the lemma core, not to arbitrary surface strings

Example:

- lemma root: `apply`
- lexical variants:
  - `applies`
  - `applied`
  - `applying`
  - `reapply`
  - `unapplied`

The semantic anchor remains `apply`.

## Symbolic Constructor Consequence

The canonical symbolic body is:

- prefix operators
- lemma core
- suffix operators

This means the root core of a canonical `SymbolicConstructorTriplet` is a lemma anchor, not a surface-word snapshot.

Merged glyphs may compress that structure visually, but they do not replace the lemma-anchored constructor body as the canonical meaning object.

## Atlas Intake Consequence

The canonical intake path is now understood as:

1. surface-language intake
2. lemma normalization
3. root lookup by lemma
4. operator recognition
5. symbolic constructor-body formation
6. `EngramDraft` binding

This keeps English or other language input as an intake surface rather than the canonical meaning substrate itself.

## Cross-Language Rule

Cross-language mapping remains future work, but the constitution already fixes the rule:

- cross-language lexical forms may map into one canonical lemma anchor only through explicit normalization policy
- no heuristic best-match cross-language rooting is allowed by default

That preserves the repo’s non-heuristic atlas posture.

## Ambiguity Rule

When surface input can map to more than one plausible lemma identity:

- the system must not silently choose one canonical root
- the result remains:
  - `NeedsSpecification`
  - or another later lawful ambiguity state

Ambiguity is preserved until resolved, not flattened into convenience.

## Immediate Consequence

The atlas now has a stronger root-core reading:

- constructor law is anchored to lemma identity
- lexical variants are compatible expressions, not rival roots
- later Engram formation gets one stable root anchor instead of many surface-word candidates

This is the necessary bedrock before broader seed `GEL` contextualization begins.
