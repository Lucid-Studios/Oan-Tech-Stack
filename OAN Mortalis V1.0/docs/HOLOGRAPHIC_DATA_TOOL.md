# HOLOGRAPHIC_DATA_TOOL

## Purpose

This document defines the Holographic Data Tool as a governed artifact system for creating `.hopng` objects.

The goal is not merely to store images.

The goal is to create lawful, signed, inspectable carriers that can hold layered meaning and later support higher-order forms of relation, temporality, identity, commitment, and runtime participation.

Assessment date:

- March 6, 2026

## Core Definition

The Holographic Data Tool creates `.hopng` artifacts:

- visible PNG-based carriers
- backed by structured sidecars
- governed by trust, validation, and relational law

`.hopng` should therefore be treated as a governed artifact object, not as a flat media file.

## Governing Intent

The system is being built to support:

- trusted artifact existence
- lawful relationality between artifacts
- inspectable structure
- controlled semantic growth
- future identity-bearing and commitment-bearing forms
- later runtime participation inside Sanctuary and OE life

## Artifact Structure

A `.hopng` object is expected to combine:

- a visible PNG-oriented carrier surface
- structured sidecars for manifest, trust, validation, and relational metadata
- lawful projection bindings across Theater-local meanings where applicable

The carrier is not the whole artifact.

The artifact includes the governed packaging, sidecar structure, and validation posture that make it lawful.

- governed holographic artifact system

Primary purpose:

- create lawful symbolic carriers with visible, structural, and later semantic depth

Authority status:

- non-authoritative carrier and packaging system by default
- may carry governed material without becoming the authority that defines that material

Identity relation:

- identity-neutral by default
- may later host identity-bearing payloads only through explicit governed admission

Mutation rule:

- may package, validate, compare, and relate artifacts lawfully
- must not silently collapse Theater-local projections into one flat interpretation
- must not silently promote privileged or cryptic structure into Prime-safe release form

## Theater Relation

`.hopng` objects are compatible with multi-Theater projection.

Illustrative projection families:

- visible projection -> Prime Theater

The artifact must preserve the lawful gluing relation between these projections.

Cross-Theater interpretation of a `.hopng` object is valid only when:

- the projection boundary is explicit
- the governing relation is inspectable
- required invariants survive transition

## Trust And Inspection

The current trust model already distinguishes:

- Prime-safe inspection
- privileged inspection

That distinction should remain constitutional.

Inspection visibility must not imply unrestricted mutation or release rights.

## Bounded Build Integration

The active build may reference the Holographic Data Tool as a bounded toolchain dependency for `.hopng` validation and inspection.

Current integration posture:

- optional only
- local and operator-scoped only
- never constitutive of Sanctuary founding, bonding, re-entry, or OE binding in v0.1

The approved build path is:

- root `build.ps1` and `test.ps1` may run optional `.hopng` validation through `tools/verify-hopng-toolchain.ps1`
- HDT discovery must come from:
  - `-HdtRoot`
  - `HDT_ROOT`
  - ignored local config under `.local/`
- no tracked file may hard-code an external HDT root

The optional validation lane may use HDT for:

- `.hopng` artifact validation
- Prime-safe inspection
- artifact surface comparison

It must not use HDT in this phase for:

- Sanctuary founding gates
- re-entry gates
- constitutive OE binding
- runtime authority or continuity truth

That boundary remains in force until later-phase commands such as `Invoke-HOPNGFormation` and `Bind-HOPNGToOE` are implemented and explicitly admitted by runtime law.
