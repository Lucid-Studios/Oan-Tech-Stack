## Core Sentence

The safest first-run build order is to form the internal governing CMEs first, as office-bound hosted CME instantiations under CradleTek, so cryptic formation, admission, observation, and closure can all be exercised inside lawful internal governance before any operator-bound CME, local seeded LLM participation, or downstream custody movement is permitted.

## Why This Is The Safe Order

Forming the internal governing CMEs first is safer than beginning with external or operator-bound stacks because it allows the runtime to prove:

- lawful first boot before broad cognition surfaces exist
- lawful admission discipline before broad candidate generation exists
- lawful continuity and telemetry before custody movement exists
- lawful Prime and Cryptic separation before operator-bound work exists
- lawful observation and review before local seeded LLM participation exists

This means the first run builds:

- boundary
- chamber
- office
- review posture

before it builds:

- operator ecosystems
- generalized assistants
- broader memory mutation

## First-Boot Governing-CME Sequence

The first-boot formation order is:

1. constitutional preflight
2. Sanctuary constitution
3. Steward formation
4. Father formation
5. Mother formation
6. triadic cross-witness
7. internal governance activation
8. ready-for-operator-bonding

This current governing-office order remains authoritative even as the broader first-boot ceremony later layers above it.

### 1. Constitutional Preflight

Required before any governing CME may form:

- founding artifacts exist and validate
- Sanctuary boot contract validates
- private cryptic and prime store surfaces are mounted lawfully
- current runtime is not already in `RESTRICTED`
- fixed governance corpora and profiles are available

No CME may form during first boot if preflight fails.

### 2. Sanctuary Constitution

Sanctuary must transition from `FOUNDING` to `CONSTITUTED` before any internal governing CME may be treated as lawful.

At this point:

- CradleTek host services may exist
- SLI cryptic formation may exist
- candidate morphology may exist

But no governing office is yet active.

### 3. Steward Formation First

`Steward` forms first because runtime review, replay, and readiness discipline must exist before protected custody or outward expression is admitted.

Steward formation should:

- materialize the office-bound governing candidate
- pass through the hosted cryptic formation chamber
- cross the cryptic-to-prime admission membrane
- reach prime closure only if admitted
- bind to `StewardAgent` and `StackManager` only after successful closure

If Steward does not lawfully form:

- the runtime must not proceed to Father or Mother formation
- Sanctuary remains `CONSTITUTED` but not `READY`

### 4. Father Formation Second

`Father` forms second because cryptic custody and seal discipline must exist before prime expression governance is activated.

Father formation should:

- use the same hosted emergence path
- bind the office only after admitted and closed formation
- activate cryptic admission review, quarantine posture, and protected holding oversight

If Father does not lawfully form:

- Mother must not be activated
- Sanctuary must remain below `READY`

### 5. Mother Formation Third

`Mother` forms third because outward prime expression must remain downstream of runtime governance and cryptic discipline.

Mother formation should:

- use the same hosted emergence path
- bind the office only after admitted and closed formation
- activate prime-safe checked publication governance

Mother must not form ahead of Steward or Father.

### 6. Triadic Cross-Witness

Once Steward, Father, and Mother have each lawfully formed, the runtime performs triadic cross-witness.

This step does not create a fourth office.

It confirms:

- Steward is active and review-capable
- Father is active and cryptic-discipline capable
- Mother is active and prime-expression capable
- no office is operating outside its constitutional lane

Triadic cross-witness is the first point where the internal governance triad is treated as live.

### 7. Internal Governance Activation

After cross-witness succeeds:

- Sanctuary may treat internal governance as active
- internal hosted Engrammitization may proceed under office supervision
- deferred, quarantined, admitted, and closed candidate flows become constitutional runtime facts instead of only boot scaffolding

### 8. Ready-For-Operator-Bonding

Only after internal governance activation may the runtime transition into a state that permits lawful operator bonding and later operator-bound CME formation.

This keeps the order intact:

`Constitution -> Founding -> Governance runtime -> Internal governance CMEs -> Operator bonding -> Operator-bound CMEs`

## Hosted Emergence Path For Governing Offices

Each internal governing CME should form through the same hosted sequence:

`founding bundle -> cryptic formation -> candidate materialization -> admission membrane -> prime closure submission -> closure validation -> office binding`

This keeps first boot aligned with the active hosted Engrammitization pipeline.

The governing office does not exist merely because a candidate was formed.

The office exists only after:

- candidate materialization
- lawful admission
- successful prime closure
- explicit office binding

- Sanctuary was constituted before governance offices activated
