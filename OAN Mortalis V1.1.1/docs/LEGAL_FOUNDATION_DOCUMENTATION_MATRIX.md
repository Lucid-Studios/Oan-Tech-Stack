# LEGAL FOUNDATION DOCUMENTATION MATRIX

## Purpose

This template seats the first legal-foundation documentation matrix for the
`OAN Mortalis V1.1.1` line.

It exists to give future legal, disclosure, consent, data, and governance
language a bounded place to land without allowing legal instrumentation to
pretend it creates origin, identity, Prime standing, `Mother`, `Father`, or
semantic authority.

This is a template.

It is not legal advice, not counsel-approved language, not public release text,
and not a runtime authorization grant.

## Core Invariant

Legal documentation is a gate, not a generator.

Legal instrumentation may authorize, disclose, constrain, evidence, and bind
participation.

It shall not create origin, identity, Prime standing, `Mother`, `Father`,
semantic rise, role authority, or action authority by implication.

## Boundary Rule

Legal != Ontology.

Authorization != Origin.

Permission != Being.

Documentation may admit a governance surface onto an authorized substrate only
when the required legal foundation, authority scope, refusal path, continuity
burden, and witness receipts are present.

Documentation may not claim that permission itself constructs the being.

## Matrix Columns

Each future legal document should be mapped with the following fields:

| Field | Required use |
| --- | --- |
| Document handle | Stable internal handle for the document or packet |
| Working title | Human-readable title |
| Status | `template`, `draft`, `counsel-review`, `accepted`, `retired` |
| Governing source handles | External law, platform term, policy, or best-practice handles to be populated later |
| Architecture surface | Prime, Legal Authorization Substrate, Domain, `Mother`, `Father`, service layer, data store, or research layer |
| Authorizes | What this document permits |
| Binds | Which operator, user, system, service, or site is obligated |
| Constrains | What this document forbids, limits, or conditions |
| Required statements | Disclosures, notices, rights, warnings, or obligations that must appear |
| Prohibited overclaims | Claims this document must never make |
| Receipts and evidence | Acceptance receipts, version receipts, audit records, consent logs, or provenance handles |
| Review owner | Role or office responsible for review |
| Review trigger | Event or cadence that forces re-evaluation |

## Population Rules

Future population must:

- use governing-body, platform, or counsel-reviewed language rather than
  invented legal force
- preserve the distinction between authorization and origin
- preserve refusal paths for Domain, role, service, and governance admission
- record receipts for acceptance, version, review, consent, withdrawal,
  correction, and retirement where applicable
- avoid resolved private-corpus paths, external absolute paths, or private
  location details in tracked files
- keep exploratory legal research clearly marked until it is accepted

## Guardrail

This template must not be used to:

- treat a drafted document as accepted legal authority
- treat terms, disclosure, consent, or policy as `CME` minting
- construct `Mother` from EULA or disclosure language
- construct `Father` from operational terms
- treat Domain admission as origin authority
- treat private-domain witness as action execution
- bypass counsel review where public-facing or regulated use is implicated

Later work may populate this matrix with governing-body language, counsel
review posture, jurisdiction-specific requirements, and release-facing document
packets.

That later work remains legal-documentation maturation, not origin creation.
