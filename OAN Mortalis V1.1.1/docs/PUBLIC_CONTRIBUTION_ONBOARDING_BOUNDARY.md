# Public Contribution Onboarding Boundary

## Purpose

This document defines the public contribution entry boundary for the active
`OAN Mortalis V1.1.1` line.

It inherits the public encounter law in
`PUBLIC_ENCOUNTER_BOUNDARY_AND_NON_CLAIMS.md` and applies it to contributors,
issue authors, pull request authors, reviewers, and external analysis.

Encounter answers how the repository may be seen.
Entry answers how outside participation may begin without importing false
identity, authority, certainty, or governance standing.

## Entry Statement

Contributions enter as proposed changes to a governed engineering workspace.

They do not enter as:

- identity claims
- personhood claims
- autonomous authority
- legal authority
- custody authority
- proof of `CME` standing
- proof of installer completion
- proof that private or hosted runtime surfaces are present in the public
  checkout

Every contribution remains subordinate to repo-local governance, executable
truth, audit evidence, and maintainer review.

## Required Contributor Posture

Contributors must treat the repository as a lawfully bounded engineered
cognition framework.

Before proposing a change, contributors should read:

- `AGENTS.md`
- `README.md`
- `CONTRIBUTING.md`
- `OAN Mortalis V1.1.1/docs/PUBLIC_ENCOUNTER_BOUNDARY_AND_NON_CLAIMS.md`
- `OAN Mortalis V1.1.1/docs/BUILD_READINESS.md`

Contribution language must distinguish:

- implemented and verified behavior
- contract-backed doctrine
- documentation-only framing
- held or deferred work
- exploratory horizon language

Exploratory language may be proposed.
It may not be presented as current executable truth.

## Allowed Entry Paths

Public participation may enter through:

- issues
- pull requests
- review discussion
- documentation clarification
- bounded bug reports
- test failures with reproduction notes
- private disclosure channels for security, conduct, legal, or sensitive
  material

Public participation must not enter by posting:

- private corpus paths
- credentials or tokens
- machine-local secrets
- private runtime payloads
- hosted seed `LLM` payloads
- sensitive legal, identity, custody, or operator material
- external absolute paths that reveal private local structure

## Governance Boundary

Public contribution does not grant governance authority.

Maintainers may receive, evaluate, defer, narrow, or refuse a contribution.
That refusal is part of the repository boundary, not a failure of participation.

`Build Contracts/` remains a controlled governance surface.
Public changes to that surface require explicit governance scope and must not be
mixed with ordinary implementation work.

Active implementation work defaults to `OAN Mortalis V1.1.1/` unless the task
explicitly requires a governed contract or public-boundary change.

## Domain Role Capacity Boundary

Contribution authority is scoped through:

```text
Domain -> Role -> Capacity
```

Opening an issue grants no write capacity.
Opening a pull request grants no merge authority.
Reviewing a pull request grants no governance authority.
Using stack vocabulary grants no ontology, custody, legal standing, or `CME`
standing.

Maintainers remain responsible for deciding whether a contribution is admitted,
revised, deferred, or refused.

## Evidence And Verification

Every meaningful contribution should state:

- what changed
- why it changed
- which architecture layer is affected
- how it was verified
- what it does not claim

Evidence may include tests, build logs, reproduction steps, documentation
citations, or audit witnesses.

Evidence does not become authority merely by being present.
It remains bounded, contextual, and revisable.

## Non-Claims

Public contribution must not claim or imply:

- contributor identity becomes stack identity
- contribution language creates personhood
- issue discussion creates governance law
- pull request authorship creates custody
- review approval creates legal authority
- documentation wording creates `CME` standing
- tests alone create ontology
- public repo access proves installer completion
- hosted seed `LLM` behavior is shipped in the public checkout
- exploratory doctrine is current executable truth

## Onboarding Consequence

Contributor-facing surfaces must route public entry through this boundary.

If a later guide, issue template, pull request template, release note, or public
summary invites outside participation, it must preserve the public encounter
invariant:

No external statement may imply more identity, authority, or certainty than the internal system can lawfully support.
