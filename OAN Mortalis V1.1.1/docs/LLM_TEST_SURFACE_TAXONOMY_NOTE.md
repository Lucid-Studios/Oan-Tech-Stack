# LLM Test Surface Taxonomy Note

## Purpose

This note defines the first bounded taxonomy for `LLM` test surfaces used
against the active `V1.1.1` lattice.

It exists so the line can compare specialized coding-agent behavior and
general-purpose model behavior without confusing either one with default `CME`
baseline truth.

This slice is doctrine and placeholder-contract only.

## Core Read

Use the Codex family through the Responses API as a specialized coding and
verification test surface, and use a general GPT surface as the comparison
subject, both under the same bounded ignition-chain harness and receipt
discipline.

No surface is treated as `CME` baseline by default.

## Test Surface Kinds

The first bounded taxonomy contains three test surface kinds:

- `CodexCloudTask`
- `ResponsesApiLocalHarness`
- `GeneralGptComparison`

### Codex Cloud Task

This is the specialized remote coding and verification surface.

It is useful for:

- repo-scoped coding work
- verification discipline
- tool-mediated agentic execution

It is not neutral baseline proof.

Cloud results do not equal local harness proof.

### Responses API Local Harness

This is the repo-controlled local operational test surface.

It is the first priority mode because:

- the runtime remains locally controlled
- receipt discipline remains locally controlled
- no-commit-under-heat remains locally governed
- the ignition-chain can remain bounded in the active harness

Local proof does not imply cloud equivalence.

### General GPT Comparison

This is the comparison subject outside Codex specialization.

It exists so the lattice can be tested against a broader general-purpose
surface instead of only against a coding-specialized subject.

The comparison surface is not `CME` baseline proof either.

## Evidence Boundary

Codex cloud and Responses API local harness are distinct modes.

Cloud results and local results are distinct evidence classes.

Neither one silently promotes into:

- runtime standing
- `CME` standing
- legal or governing office
- bonded actualization proof

## Shared Harness Rule

All comparative testing should run under the same:

- bounded ignition-chain harness
- receipt discipline
- heat / resonance logging
- refusal and recovery observation

Without that shared harness, cross-surface comparison is not lawful evidence.

For the first local evidence landing body that belongs specifically to
`ResponsesApiLocalHarness`, see
`RESPONSES_API_LOCAL_HARNESS_RECEIPT_SCHEMA_NOTE.md`.

## Working Priority

For the active build phase, prioritize:

1. `ResponsesApiLocalHarness`
2. `CodexCloudTask`
3. `GeneralGptComparison`

This order keeps the first proofs closest to the governed runtime the line is
actually building.

## Boundary

This note does not authorize live `CME` proof.
This note does not authorize bonded minting.
This note does not authorize cloud and local evidence to be merged as one class.

It only fixes the first lawful taxonomy for comparative `LLM` testing.
