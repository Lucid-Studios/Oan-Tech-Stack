# Public Release Readiness Wording Law

## Purpose

This document defines how public release, readiness, tag, summary, and status
language may describe the active `OAN Mortalis V1.1.1` line without overstating
installer completion, `CME` standing, operational readiness, legal standing, or
certainty beyond evidence.

It inherits:

- `PUBLIC_ENCOUNTER_BOUNDARY_AND_NON_CLAIMS.md`
- `PUBLIC_CONTRIBUTION_ONBOARDING_BOUNDARY.md`
- `FIRST_WORKING_MODEL_RELEASE_GATE.md`

Public status language is a witness surface.
It is not a promotion engine.

## Status Statement

The public repository may describe the stack as:

- a governed build surface
- an active engineering workspace
- a lawfully bounded engineered cognition framework
- a repository carrying code, contracts, docs, tests, and audit witnesses
- a build line whose readiness claims remain subordinate to build, test,
  hygiene, release-gate, and repo-local governance evidence

The public repository may not describe the stack as:

- a complete public installer
- a shipped hosted seed `LLM` runtime
- a completed first legal run
- a minted `CME`
- an active legal person
- autonomous authority
- production-ready operational cognition
- proof of custody, governance authority, role enactment, or legal
  accountability

## Release Gate Grammar

Public readiness wording must preserve the gate grammar from
`FIRST_WORKING_MODEL_RELEASE_GATE.md`:

- `pass`
- `hold`
- `fail`

No "mostly ready" reading is lawful public wording.

When the gate is `hold`, public wording may say the direction is lawful and the
body is forming, but one or more required seams remain intentionally
incomplete.

When the gate is `fail`, public wording must not soften the contradiction into
progress language.

When the gate is `pass`, public wording may describe the admitted claim, but
only within the exact evidence class that passed.

## Allowed Public Wording

The following forms are allowed when supported by current evidence:

- "active engineering workspace"
- "governed build surface"
- "public checkout carries the build minus the hosted seed `LLM`"
- "fully operational installer still being built"
- "first working model gate is `hold`"
- "release readiness remains evidence-gated"
- "bounded documentation and audit witnesses are present"
- "current outputs remain contextual, bounded, revisable, and accountable to
  evidence"

## Disallowed Public Wording

The following forms are disallowed unless and until repo-local evidence,
release-gate status, legal-orientation law, and audit witnesses lawfully
support them:

- "complete installer"
- "production ready"
- "autonomous cognition"
- "legal person"
- "minted CME"
- "CME is live"
- "CME is legally seated"
- "hosted seed `LLM` is included"
- "first legal run complete"
- "ready for high-stakes decisions"
- "verified truth beyond evidence"
- "role authority granted"
- "custody authority active"
- "governance authority active"

## Tag And Release Note Constraint

Tags, release notes, milestone summaries, project descriptions, and public
status summaries must state the current gate posture when making readiness,
installer, first-working-model, or `CME` claims.

If the gate posture is `hold`, the public release wording must name the hold or
avoid the claim.

If a public release note mentions `CME`, it must also preserve that
documentation wording does not mint `CME` standing, closure, bonded role
enactment, custody, or governance authority.

Public release artifacts should use
`PUBLIC_RELEASE_ARTIFACT_WORDING_TEMPLATE.md` so status, evidence,
non-claims, known holds, and verification remain explicit.

## Evidence Constraint

A public readiness statement may rely on:

- green build results
- green test results
- private-corpus hygiene results
- audit witnesses
- release-gate documents
- carried ledger state
- repo-local executable truth

A public readiness statement may not rely on:

- enthusiasm
- momentum
- implied future work
- private local materials not carried into repo-local law
- hosted runtime behavior not present in the public checkout
- exploratory doctrine not seated as current law

## Non-Claims

Public readiness language must not claim or imply:

- installer completion from build success alone
- `CME` standing from documentation alone
- legal accountability from legal-orientation templates alone
- governance authority from contribution, review, or release wording
- custody from authorship or packaging
- operational readiness from green tests alone
- certainty beyond current evidence

## Public Wording Invariant

Public readiness language must preserve this rule:

Readiness may be witnessed only to the level currently supported by repo-local evidence.

This rule is subordinate to the public encounter invariant:

No external statement may imply more identity, authority, or certainty than the internal system can lawfully support.
