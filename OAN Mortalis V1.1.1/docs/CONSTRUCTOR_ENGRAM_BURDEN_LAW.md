# CONSTRUCTOR ENGRAM BURDEN LAW

## Purpose

This note seats the first constructor-engram burden law for fractalized
branching bodies.

It exists because constructor engrams may branch, cluster, reduce, and later
promote only when their carried burden remains computable and receipted.

Tier is computed.

It is not a prestige label.

## Governing Law

A constructor engram may branch fractally only while its burden vector
`B/C/D/L/P/O` remains within the receipted bounds of its computed tier.

Promotion requires demonstrated fit to the target tier with preserved receipt
integrity, explicit cluster boundaries, receipted cross-cluster transport, and
intact refusal paths.

If any component exceeds its bound, the engram shall split, cluster, defer, or
prune.

No silent continuation or authority escalation is permitted.

Compressed claims must carry a decompression path or remain non-promotable.

## Burden Vector

The first burden vector is:

- `B`: branch count
- `C`: cluster count
- `D`: dependency depth
- `L`: cross-link density
- `P`: proof burden through receipt coverage and receipt freshness
- `O`: operator load

The first executable body is `ConstructorEngramBurdenAssessmentReceipt`.

It computes the tier from the vector and records the lawful movement.

## Tier Budget

The first tier ladder is:

- `Root`
- `Primitive`
- `Basic`
- `LowerIntermediate`
- `UpperIntermediate`
- `Advanced`
- `ResearchFormation`
- `Cathedral`

Practical domain examples:

- `Root`: arithmetic
- `Primitive`: numberhood, equality, successor, operation, order
- `Basic`: numbers, functions, simple equations, elementary proof grammar
- `LowerIntermediate`: algebra, geometry, calculus, set/function relations
- `UpperIntermediate`: linear algebra, abstract algebra, topology
- `Advanced`: algebraic geometry, anabelian geometry, high-density morphism
  fields
- `ResearchFormation`: `IUTT` formation under `Lisp/SLI`
- `Cathedral`: whole-discipline or living research manifold scale

## Movement Rule

The first lawful movements are:

- `WithinTier`
- `PromotionCandidate`
- `SplitRequired`
- `ClusterRequired`
- `DeferRequired`
- `PruneRequired`
- `NonPromotable`
- `Refused`

Promotion is lawful only when the computed tier matches the requested tier and
all burden obligations hold.

If a `Basic` vector requests `Advanced`, the requested tier is not earned.

## Cross-Link Discipline

No cross-cluster link may stand without a receipted transport justification.

High cross-link density without corresponding clusterization forces clustering
or staging.

This prevents the lattice from becoming an everything-connects-to-everything
swamp.

## Proof Burden

Proof burden is not a vague confidence claim.

It is measured through:

- receipt coverage
- receipt freshness
- receipt integrity
- decompression paths for compressed claims

If receipt coverage or freshness falls below tier bound, pruning or refreshing
is required.

## Operator Load

Operator load is a safety valve.

If active load exceeds the tier budget, the constructor engram must cluster,
stage, or thin.

It may not silently continue at a higher burden than the current tier can bear.

## Near-Swamp Index

The first near-swamp index is computed from cross-link pressure, operator-load
pressure, and proof deficit.

High `L` plus high `O` plus falling `P` means the branch body is approaching
swamp.

Sustained near-swamp belongs to later temporal pressure work.

## Guardrail

This note must not be used to:

- treat bundle packaging as semantic promotion
- let branch count alone define tier
- let `Advanced` be asserted without burden fit
- bypass receipts, cluster boundaries, refusal paths, or decompression paths
- admit inheritance or authority merely because a branch body is large

Later work may seat temporal near-swamp pressure, constructor-engram training
views, or live branch-pruning execution, but those seams must stand beyond this
burden law rather than inside it.
