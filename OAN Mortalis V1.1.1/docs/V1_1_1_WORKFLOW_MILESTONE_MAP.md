## Workflow Order

The intended workflow order is:

1. `M0` standalone line
2. `M1` carry-forward refinement split
3. `M2` contract-first unlock
4. `M3` root dispatch and federation control plane
5. `M4` seeded-governance build admission seam
6. `M5` runtime readiness chain
7. `M6` bounded full-body enrichment
8. `M7` source-bucket request/return loop as needed during enrichment
9. `M8` executable candidate
10. `M9` production-pre-release form
11. `M10` seed-LLM wrinkle-test pause
12. `M11` release handshake and ratification
13. `M12` publishable master thread
14. `M13` post-publish governance loop

This order is not strictly linear in every local detail.

Some evidence surfaces already exist ahead of later admissions.

The governing rule is:

- evidence may be prepared early
- admission and widening must still occur in lawful order

## Non-Goals

This workflow map does not:

- widen live `SoulFrame`, `AgentiCore`, `ListeningFrame`, or tool-use authority
- collapse research or publication doctrine directly into executable truth
- let source-bucket output mutate Build without receipts
- skip the seed-LLM pause seam
- replace operator-governed release and ratification
