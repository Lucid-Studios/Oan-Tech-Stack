# Public GitHub Entry Template Boundary

## Purpose

This document defines the public GitHub entry template boundary for issues,
feature requests, and pull requests.

It applies the public contribution onboarding boundary at the exact point where
outside participation enters the repository.

The templates are intake surfaces.
They are not promotion engines, authority grants, custody grants, release
claims, or `CME` standing claims.

## Required Intake Questions

GitHub entry templates must ask for enough information to preserve boundary
truth before maintainers evaluate the work.

The required intake classes are:

- affected architecture layer
- affected stratum or boundary type
- evidence, witnesses, tests, or reproduction steps
- non-claims made by the report, request, or pull request
- standing, governance, authority, readiness, or completion risk
- private-path and private-corpus hygiene confirmation

## Layer And Stratum Framing

Public GitHub entry must make contributors choose or name the layer they think
they are touching.

The allowed broad strata are:

- Constitutional or governance
- Infrastructure or verification
- Runtime or executable behavior
- Documentation or doctrine
- Public boundary
- Other bounded stratum

This choice is not self-authorizing.
It is a contributor hypothesis for maintainer review.

## Non-Claim Requirement

Every pull request must state what the change does not claim when the change
could be mistaken for:

- identity
- custody
- governance authority
- legal authority
- `CME` standing
- role enactment
- installer completion
- production readiness
- certainty beyond evidence

Issues and feature requests must provide the same non-claim note when their
wording could be mistaken for standing, authority, readiness, or completion.

## Witness Requirement

Pull requests must name the tests, audit witnesses, build commands, hygiene
checks, documentation citations, or reproduction evidence used to support the
change.

Bug reports must ask for reproduction steps and sanitized evidence.

Feature requests must ask for expected witnesses or verification surfaces.

Evidence remains bounded, contextual, and revisable.
It does not create authority merely by being present.

## Standing And Readiness Risk

Templates must require explicit attention when a change could be mistaken for:

- `CME` minting
- legal personhood
- autonomous authority
- governance authority
- custody
- role enactment
- hosted seed `LLM` shipment
- complete installer readiness
- production readiness
- high-stakes decision readiness

When this risk exists, the entry must say how the public encounter,
contribution onboarding, and release readiness wording laws are preserved.

## Current Entry Surfaces

The current public GitHub entry surfaces are:

- `.github/pull_request_template.md`
- `.github/ISSUE_TEMPLATE/bug_report.yml`
- `.github/ISSUE_TEMPLATE/feature_request.yml`
- `.github/ISSUE_TEMPLATE/config.yml`

These surfaces must point contributors toward the active `V1.1.1` build and
its current public boundary law, not retired `V1.0` paths.

## Boundary Invariant

GitHub entry templates must preserve this rule:

Template completion is not admission.

Template completion does not grant identity, custody, governance authority,
legal authority, `CME` standing, release readiness, installer completion, or
certainty beyond evidence.

No external statement may imply more identity, authority, or certainty than the internal system can lawfully support.
