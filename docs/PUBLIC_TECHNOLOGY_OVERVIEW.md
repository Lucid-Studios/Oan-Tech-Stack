# Public Technology Overview

OAN technology is a governance-oriented approach to building accountable
AI-assisted systems. The public claim is intentionally narrow: complex
AI-assisted systems need explicit boundaries for authority, continuity, review,
disclosure, safety, and human responsibility.

The private engineering program explores how those boundaries can be
represented inside working software. This public repository does not publish
that software, the private build, or a reconstruction path.

## Public Access Posture

The public access posture is citizen-science oriented. Readers may use this
repository for education, stewardship literacy, release-boundary review, and
public accountability. Future operational participation would require separate
education, certification, approval, and stewardship review.

Public access does not mean build access, runtime access, private corpus
access, or authority to operate a private system.

## Design Principles

- Human authority remains primary.
- System claims should be bounded by evidence.
- Sensitive continuity, memory, and identity-adjacent material require custody
  controls.
- Public descriptions should not become build recipes.
- Safety review should occur before publication, deployment, or operational
  expansion.
- Public education should improve understanding without releasing private
  capability.
- Familiarity with a system should not be confused with personhood, legal
  authority, or identity custody.
- Model capability should not be collapsed into continuity, custody, or
  operational authority.

## Custodial Cognition Safety

The current public safety frame is custodial cognition safety. In plain terms,
the project treats advanced AI-assisted systems as requiring structural
separation between:

- model capability and authority;
- interaction history and custody;
- familiar behavior and personhood claims;
- telemetry and surveillance;
- public explanation and private implementation.

This is a public safety posture, not a publication of internal mechanisms.

## Familiar But Not Human

OAN public language permits the idea that a governed system can become familiar
through continuity, reliability, shared work, and recognizable posture. That
does not make the system human, conscious by assertion, legally empowered, or
authorized for personified operation.

This boundary allows useful long-form collaboration while keeping claims
evidence-limited and reviewable.

## End-User Surfaces

Publicly, an end-user Mini CME surface can be understood as a consent-based
application endpoint. It may provide model-assisted interaction, approved tool
access, avatar or presentation features, and local device adaptation across
wearable, phone, tablet, desktop, or other approved endpoint classes.

The endpoint remains a bounded interface. It does not own continuity, custody,
identity authority, protected memory, or personification standing.

## Public Non-Claims

This repository does not claim:

- production readiness;
- deployment availability;
- release of source code;
- disclosure of private architecture;
- disclosure of datasets, model payloads, or runtime payloads;
- grant of governance, custody, legal, or identity authority.

## Public Use

Readers may use this repository to understand the release posture and review
principles. They should not treat it as an implementation guide.

For an academic-style review of the current public claim boundary, see
