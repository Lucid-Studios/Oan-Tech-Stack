# SERVICE LIFT PRECONDITIONS AND REFUSAL LAW

## Purpose

This note defines when hosted motion for the seated Prime/Cryptic braid is
even discussable in `V1.2.1`.

It exists so passive residency, seated working surfaces, and runtime desire do
not get over-read into lawful hosted motion.

## Governing Compression

> hosted residency is not hosted activation.

> service lift changes hosting posture, not governing identity.

> no hosted motion may be treated as lawful unless the preconditions,
> authority surface, supervision binding, and receipt chain are all present
> together.

## Required Preconditions

Hosted motion is discussable only when all of the following are already
legible and receipted:

- the release gate still visibly governs the seam
- issued `SanctuaryID.RTME` handle burden is already legible
- hosted cryptic bundle residency is already seated
- canonical floor-set readiness is already witnessed
- Prime/Cryptic supervisory posture is already bounded
- install-assent and identity footing already sits below this seam and does
  not replace it
- `CmePlacementWithheld` still remains explicit active law

- `RtmeServiceLiftPreconditionSnapshot`

## Refusal And Withhold Rule

Hosted motion must be refused or withheld when any required condition is:

- absent
- implied rather than receipted
- borrowed from nearby presence
- overstated as active governing standing

The bounded passive refusal family for this read now lives in:

- `ServiceLiftRefusalReason`

The lawful response to missing or implied lift conditions is:

- preserve the prior disabled posture
- preserve `CmePlacementWithheld`
- require explicit authority rather than convenience inference
