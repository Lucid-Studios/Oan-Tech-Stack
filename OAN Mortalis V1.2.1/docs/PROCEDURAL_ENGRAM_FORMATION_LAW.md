# PROCEDURAL_ENGRAM_FORMATION_LAW

## Purpose

This note defines the first lawful formation of procedural engrams inside
`Sanctuary.GEL`.

It exists to fix how method-bearing semantic structures derive from
propositions without collapsing into execution or runtime law.

## Governing Compression

> procedures tell `GEL` how it must behave toward that seeming.
>
> procedural engrams define handling method, not execution authority.

## Formation Preconditions

A procedural engram may form only when:

- one or more propositions already stand
- the method office is explicit
- preconditions are explicit
- refusal conditions are explicit
- witness is present

Procedural engrams may never derive directly from raw ingress.

## Constructor Eligibility

Procedural engrams may stand only at:

- `Basic`
- `Intermediate`
- `Advanced`
- `Master`

Minimal procedural refusal/check engrams may stand at `Basic`.
Richer method bundles begin at `Intermediate`.

## Procedure Offices

The minimal procedure offices admitted in this phase are:

- `discernment`
- `refusal`
- `transformation`
- `continuity`
- `correspondence`

## Canonical Fields

Every procedural engram should carry at minimum:

- `procedure_id`
- `engram_id`
- `constructor_class`
- `categorical_class = procedural`
- `proposition_refs[]`
- `method_class`
- `preconditions[]`
- `refusal_conditions[]`
- `allowed_effect_class = NON-RUNTIME`
- `continuity_refs[]`
- `witness_refs[]`

## Refusal Rules

The following must force refusal, hold, or non-formation:

- direct derivation from ingress
- missing proposition references
- implied execution authority
- missing refusal conditions
- borrowed runtime effect

## Explicit Non-Grants

This note does not grant:

- active execution
- runtime law
- service authorization
- predicate promotion
- engram minting

## Working Summary

`Sanctuary.GEL` now fixes the first lawful method-bearing engrams.

Those engrams derive from propositions, govern handling posture, and remain
non-runtime in this phase.
