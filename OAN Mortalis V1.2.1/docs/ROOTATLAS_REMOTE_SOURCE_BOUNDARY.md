# ROOTATLAS_REMOTE_SOURCE_BOUNDARY

## Purpose

This note defines the lawful boundary between remote `RootAtlas` source
authority and local `V1.2.1` install reality.

It exists to stop three false assumptions:

- that `RootAtlas` is locally installed with the product
- that ordinary runtime work may read research servers directly
- that local `Sanctuary.GEL` is just another name for `RootAtlas`

## Governing Compression

> `RootAtlas` lives only on Research Servers.
> No local install contains `RootAtlas`.
> Local installs receive only signed, versioned bootstrap payloads derived from
> `RootAtlas`.
> Those payloads hydrate `Sanctuary.GEL`, which is the first lawful local
> substrate.

## What `RootAtlas` Is

`RootAtlas` is:

- remote source ancestry
- remote translation-authoring medium
- remote source authority for first-install payload derivation

It is not:

- the local product data store
- the local `RTME` working surface
- a directly mounted install-side filesystem body

## Local Install Rule

The install boundary is simple:

- no local install contains `RootAtlas`
- no local install should claim direct `RootAtlas` residency
- no ordinary `EC` or local-runtime path may depend on direct research-server
  reads

The lawful install-side substrate is:

- `Sanctuary.GEL`

## Payload Rule

When a new line or chamber is first hydrated:

- source ancestry remains remote in `RootAtlas`
- a signed and versioned bootstrap payload is produced from that ancestry
- the payload is fetched and verified by the local install
- the payload hydrates `Sanctuary.GEL`

Templates, first lawful knowledge stores, and first line-local engram-bearing
surfaces therefore arrive:

- from `RootAtlas` derivation
- through payload verification
- into `Sanctuary.GEL`

They do not arrive by treating `RootAtlas` as if it were locally present.

## Non-Collapse Rules

- `RootAtlas` is not `Sanctuary.GEL`
- `RootAtlas` is not the local `RTME`
- no runtime note should describe `RootAtlas` as part of the local install body
- no ordinary runtime lane should read Research Servers directly
- remote source ancestry must arrive only through signed, versioned payloads

## Implementation Consequences

`V1.2.1` should use this note when placing:

- `Sanctuary.GEL` bootstrap law
- `SanctuaryID.RTME` service issuance
- first template and knowledge-base hydration
- later `SoulFrame` bridge and cryptic seal work

This note fixes the remote/local boundary before deeper source carry forward
begins.
