# CME IP INHERITANCE AND CREATION SCOPE LAW

## Purpose

This note defines the first scoped IP authority split for content that may be
used with a future `CME` in `V1.2.1`.

It exists so submitted content does not silently collapse into inheritance,
creation use, or identity authority.

## Governing Compression

> submitted IP does not automatically become `CME` inheritance.

> authorized use in creation does not automatically grant identity
> inheritance.

## Scope Split

The bounded passive scope records now live in:

- `InheritedIpScopeRecord`
- `CreationUseScopeRecord`
- `CmeContentAuthorityBundle`

`Inheritance-scope IP` means content that may shape:

- carried identity
- voice
- style
- memory seed
- representational footing

`Creation-use IP` means content that may be used for:

- outputs
- campaigns
- assets
- prompts
- training examples
- brand material

without becoming identity inheritance.

## Scoped Authority Rule

Every admitted IP scope must remain:

- scoped
- witnessed
- role-valid
- separate between inheritance and creation use

No generic upload or generic disclosure line may replace explicit scope.

This note does not grant:

- automatic `CME` inheritance
- governing `CME` placement
- CradleTek authorization
- runtime governing action

`V1.2.1` now fixes inheritance-scope IP and creation-use IP as separate,
witnessed, and role-valid authority lanes beneath pre-Cradle standing.
