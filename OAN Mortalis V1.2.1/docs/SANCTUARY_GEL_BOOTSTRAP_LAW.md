# SANCTUARY_GEL_BOOTSTRAP_LAW

## Purpose

This note defines `Sanctuary.GEL` as the first lawful local substrate of a
`V1.2.1` install.

It exists to keep four things distinct:

- remote `RootAtlas` source ancestry
- signed bootstrap payload
- local `Sanctuary.GEL` substrate
- later hot cryptic runtime work

## Governing Compression

> No local install contains `RootAtlas`.
> The first lawful local substrate is `Sanctuary.GEL`.
> Templates, first knowledge bases, and first local engram-bearing stores are
> hydrated into `Sanctuary.GEL` from signed bootstrap payloads.
> No lawful `RTME` wake should occur before `Sanctuary.GEL` exists.

## Office

`Sanctuary.GEL` is:

- the first lawful local data store
- the first local install-side memory and template substrate
- the first chamber-local knowledge surface from which later runtime work may
  begin

It is not:

- remote `RootAtlas`
- a generic scratch store
- a substitute for later cryptic work lanes such as `cGoA`

## Bootstrap Order

The install-first order is:

1. seal the install baseline
2. enter `Sanctuary State`
3. create and bind `Sanctuary.GEL`
4. fetch and verify the signed bootstrap payload derived from remote
   `RootAtlas`
5. hydrate `Sanctuary.GEL` with templates, first predicates, and first
   knowledge-base surfaces
6. only then permit issued `RTME` service activation

## Hydration Rule

The first hydration into `Sanctuary.GEL` may include:

- templates required for install-first chamber formation
- first lawful knowledge-base surfaces
- first engram-bearing stores needed for bounded chamber work
- bootstrap predicates and records required by later governance and braid work

These may not bypass `Sanctuary.GEL`.

## Non-Collapse Rules

- no install should claim an `RTME`-ready state before `Sanctuary.GEL` exists
- no template or knowledge-base hydration should bypass `Sanctuary.GEL`
- `Sanctuary.GEL` is the local bootstrap substrate, not the remote source
  authority
- `cGoA` and other cryptic work lanes remain later derived surfaces, not the
  first install substrate
- `Sanctuary.MoS` and `Sanctuary.cMoS` remain later storage-seat surfaces
  reached only by receipted handoff

`Sanctuary.GEL` is bootstrap substrate, not `Sanctuary.cGEL`.
`Sanctuary.cGEL` is a later cryptic build chamber, not the first install
substrate.
`Sanctuary.MoS` and `Sanctuary.cMoS` are later governance-memory seats, not
bootstrap substrate.

## Implementation Consequences

`V1.2.1` should use this note when shaping:

- install-first manifests
- first lawful template hydration
- first knowledge-base carry forward
- `SanctuaryID.RTME` service issuance
- `SoulFrame` bridge and seal prerequisites

This note gives the new line its first truthful local substrate.
