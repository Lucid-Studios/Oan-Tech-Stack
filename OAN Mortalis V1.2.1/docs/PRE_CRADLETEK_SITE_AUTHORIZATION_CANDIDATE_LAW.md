# PRE CRADLETEK SITE AUTHORIZATION CANDIDATE LAW

## Purpose

This note defines the first lawful site-bound pre-Cradle authorization
candidate in `V1.2.1`.

It exists so site, operator, disclosure, and content authority can be braided
into one explicit candidate without being mistaken for governance, placement,
or activation.

## Governing Compression

> site binding, operator training, disclosure assent, and IP scope may
> authorize pre-Cradle standing, but they do not authorize governing action.

> authorized pre-Cradle standing is a site-and-operator posture, not a
> governing `CME` event.

## Candidate Braid

The first lawful candidate braid now reads:

`InstallIdentitySetCandidate -> CoreCmeUsePostureRecord -> RtmeServiceLiftPreconditionSnapshot -> Mos/cMos readiness -> CradleTekSiteBindingProfile -> OperatorTrainingAdmissionRecord -> ToolStateAuthorizationRecord[] -> FinalCmeDisclosureAgreementBundle -> CmeContentAuthorityBundle -> PreCradleSiteAuthorizationCandidate`

The bounded passive candidate record now lives in:

- `PreCradleSiteAuthorizationCandidate`

## Candidate Boundary

The candidate must preserve:

- install identity footing
- bounded `CME`-use posture
- lift readiness snapshot
- `MoS/cMoS` readiness refs
- one site-binding profile
- one operator-training admission
- one or more tool-state authorizations
- one final disclosure bundle
- one separate content-authority bundle
- one non-grant summary
- one witness chain

The disclosure bundle and the content-authority bundle remain separate in this
slice.
