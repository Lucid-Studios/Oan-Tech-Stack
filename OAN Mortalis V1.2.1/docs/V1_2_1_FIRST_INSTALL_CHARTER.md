# V1_2_1_FIRST_INSTALL_CHARTER

## Purpose

This note defines the first lawful formation posture for
`OAN Mortalis V1.2.1`.

It is the governing note for building the new line from first install before
final working form.

## Core Rule

`V1.2.1` should not begin as a blind clone of `V1.1.1`.

It should begin as a lawful first-install line that can later receive its full
source, runtime, and production-form carry forward in governed slices.

It should also correct the umbrella family before deeper carry forward begins:

- `San.*` for the new line's umbrella composition family
- `OAN.*` reserved for the future `OAN` tool, not used early as the stack
  umbrella

It should also fix the sovereignty ladder before deeper runtime carry forward
begins:

- `Sanctuary` originates governance
- `SanctuaryID.GoA` carries governing `CME` set/manifold authority
- `CradleTekID.GoA` is cradle-local operator-bonded `CME` spawn/control
- `cGoA` is the one-operator-bonded-`CME` braid lane
- `CME` is formed only within that lane on the operator-bonded path

## Install-First Order

The working order for this line is:

`Install State -> Sanctuary State -> Sanctuary.GEL bootstrap -> SanctuaryID.RTME issuance -> SoulFrame cGoA bridge and cryptic seal prerequisites -> local law and onboarding -> first source-family carry forward -> first line-local solution graph -> next sibling project carry forward -> seam-definition batch beneath the first-working-model gate -> RTME admission shell -> first witnessed trace path -> descriptive Lisp/C# carriage schema -> pre-CME substrate clarification -> SanctuaryID.GoA governing-set clarification -> SanctuaryID.GoA governance-root and cGEL stack-map clarification -> Sanctuary.GEL semantic intake clarification -> Sanctuary.GEL interior awareness and universe-law clarification -> Sanctuary.GEL rest-state and persistence clarification -> SLI symbolic transport-form clarification -> install agreement action surface and identity footing clarification -> RTME hosted service-lift corridor clarification -> Sanctuary.MoS and Sanctuary.cMoS storage-seat and governance memory clarification -> pre-Cradle site-bound operator, tool, disclosure, and content authority clarification -> final working form`

This keeps the side-by-side line honest about what exists now and what is
still withheld.

`SLI_PRIME_CRYPTIC_DUPLEX_RESONANCE_NOTE.md` is a descriptive living-state
note, not a new install-first stage.
It records the healthy read in which `Prime` holds, `Cryptic` presses,
`SLI` harmonizes, `Compass` sounds, and the line returns to hold without
losing what has not yet resolved.

## Side-By-Side Rule

`V1.2.1` is being built beside `V1.1.1`.

That means:

- `V1.1.1` remains the active executable truth
- no archive of `V1.1.1` is required for this step
- the first working-model release gate may later admit this line as
  release-form sibling body without forcing a premature line switch
- `V1.2.1` may form its production surface without forcing a premature line
  switch

## First Install Shape

The first install before final working form should carry only:

- line-local build policy
- line-local build props
- line-local doctrine and charter notes
- line-local hygiene
- line-local release intent
- reserved audit space for later receipts
- line-local telemetry taxonomy and read-only audit schema aligned to release
  state
- the corrected umbrella-family claim for the new line
- the first passive membrane-facing source-family root
- the first minimal line-local solution graph once real source exists

It should not yet claim:

- full runtime execution
- final source-family materialization
- production completeness
- line replacement of `V1.1.1`
